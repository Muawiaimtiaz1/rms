# Codebase Structure Overview - POS System v2

This document provides a comprehensive breakdown of the project directory structure, detailing what each folder and file is responsible for and the functionalities they implement.

---

## 📂 Root Directory (`/pos-system-v2`)

- **`server.js`**
  - **Purpose:** Entry point for the backend Node.js server.
  - **Functionality:** Initializes the Express application, sets up session management (`express-session`), and maps all the API endpoints (routes) and frontend static HTML file routes (like `/dashboard`, `/admin/store-monitoring`). It starts the server on port `4000`.
- **`package.json`** & **`package-lock.json`**
  - **Purpose:** Defines the project's metadata, dependencies, and npm scripts.
  - **Functionality:** Specifies packages like `express`, `better-sqlite3`, `bcryptjs`, and defines the exact versions in `package-lock.json`.
- **`migrate_multi_store.js`**
  - **Purpose:** Database migration utility script.
  - **Functionality:** Likely used to convert an older version of the database schema (v1) into the v2 structure that supports multi-tenancy (`shop_id` columns, `shops` table).

---

## 📂 `/db` (Database Configuration & Schema)

- **`db.js`**
  - **Purpose:** Initializes the SQLite database connection.
  - **Functionality:** Uses `better-sqlite3`. It turns on WAL (Write-Ahead Logging) mode, configures foreign key enforcement, and exports the database instance for use across the app.
- **`schema.sql`**
  - **Purpose:** Contains all database table definitions.
  - **Functionality:** Creates SQL tables (`shops`, `users`, `products`, `sales`, `expenses`, `support_tickets`, `activity_logs`, etc.) with proper foreign key relationships to `shops(id)` ensuring multi-tenancy.
- **`seed.js` / `seed_analytics.js`**
  - **Purpose:** Seed scripts to populate the database with initial/dummy data.
  - **Functionality:** Creates the default Superadmin account, demo stores, and injects mock sales data (useful for testing the analytics dashboard).
- **`pos.db`**, **`pos.db-shm`**, **`pos.db-wal`**
  - **Purpose:** The actual living database files generated by SQLite.

---

## 📂 `/middleware`

- **`auth.js`**
  - **Purpose:** Protects API routes by verifying user sessions and roles.
  - **Functionality:** Exports three middleware functions: 
    - `requireAuth`: Ensures a user is logged in.
    - `requireAdmin`: Ensures the logged-in user has the `admin` or `superadmin` role.
    - `requireSuperAdmin`: Ensures only the `superadmin` can access the route (used heavily in `/api/admin` routes).

---

## 📂 `/routes` (API Endpoints & Business Logic)
Each file corresponds to a specific domain in the business logic.

- **`auth.js`**
  - **Functionality:** Handles user `/login`, `/logout`, `/me`, and a superadmin-exclusive `/forgot-password`. Resolves multi-tenant variables like checking shop subscription validity and allowed panels upon login.
- **`admin.js`**
  - **Functionality:** Exclusively for Superadmin. Fetches global stats (`/store-stats`), manages shops (`/stores`, status toggles, plan updates), and views platform-wide health and activity logs.
- **`sales.js`**
  - **Functionality:** Processes checkouts. Validates stock iteratively, performs atomic database transactions to insert `sales` and `sale_items`, deducts `products` stock, and processes partial payments (debts).
- **`products.js`** & **`brands.js`**
  - **Functionality:** Handles CRUD operations for the shop inventory. Products are linked to Brands. Products utilize "Soft Deletion" (`is_deleted` flag) to hide them from the UI without deleting foreign keys linked to past sales.
- **`expenses.js`** & **`expense-categories.js`**
  - **Functionality:** Allows a shop owner to record daily outlays (rent, utilities) and log `brand_expense_payments` to specific suppliers for COGS accounting.
- **`users.js` & `shops.js` & `subscriptions.js`**
  - **Functionality:** Manages tenant entities. Admins use these to create staff within their `shop_id`. The subscription route returns the active SaaS plan end dates for the shop UI warnings.
- **`analytics.js`**
  - **Functionality:** Aggregates a specific shop's data to power their local dashboard (daily sales, low stock alerts, top products).
- **`support.js`**
  - **Functionality:** Support ticketing mechanism between Shop Owners and the Superadmin. Includes logic for tracking `is_internal` comments meant only for Superadmin visibility.

---

## 📂 `/public` (Frontend Assets & Static Pages)

- **`login.html`**
  - **Purpose:** The main login page loaded when visiting the root path `/`.
- **`dashboard.html`**
  - **Purpose:** The main interface for Shop Admins and Users. Contains the Point of Sale Terminal, Inventory Lists, Settings, and Analytics charts.
- **`store-monitoring.html`**
  - **Purpose:** The exclusive Superadmin interface to oversee all shops, system metrics, and tickets.
- **📂 `/css`**
  - **Purpose:** Contains all cascading stylesheets defining the appearance, themes, and responsiveness of the HTML files.
- **📂 `/js`**
  - **`app.js`**: Massive vanilla Javascript file bound to `dashboard.html`. It handles frontend routing (hiding/showing panels like "POS", "Inventory", "Expenses"), making fetch requests to the `/api` routes, rendering dynamic DOM elements, managing the shopping cart state, and handling local authentication state.
  - **`store-monitoring.js`**: Specifically handles the Superadmin panel frontend logic (fetching and drawing the global stats, handling datatables for shops and support tickets).
- **📂 `/uploads`**
  - **Purpose:** Directory for temporarily/permanently storing user-uploaded media (e.g., ticket attachments, product images).
